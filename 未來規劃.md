# 未來規劃：LINE 訊號確認下單系統

## 目標

本專案未來希望從「LINE 群組交易訊息紀錄工具」擴充成「半自動交易確認系統」：

1. 在指定 LINE 群組收到交易訊號。
2. 系統解析股票代號、買賣方向、價格、金額與股數。
3. 系統對指定個人 LINE userId 發出確認訊息。
4. 使用者按下「確認下單」後，才呼叫永豐金 Shioaji API。
5. 下單結果、委託狀態與成交狀態回寫資料庫，並通知使用者。

核心原則是：**群組訊息只產生待確認單，不直接下單。**

## 預期流程

```text
LINE 群組訊息
  -> webhook 接收
  -> parseTradeMessages 解析訊號
  -> 建立 OrderIntent
  -> 推送個人確認訊息
  -> 使用者按確認或取消
  -> webhook 收到 postback
  -> 檢查權限、期限、價格與風控
  -> 呼叫 Shioaji 下單
  -> 回寫委託結果
  -> LINE 通知個人
```

## 階段規劃

### Phase 1：確認單 MVP

先不接 Shioaji，完成「收到群組訊號後，推送個人確認訊息」。

待辦：

- 新增 `OrderIntent` model。
- 從 `recordTradeMessage` 拆出獨立的交易訊號處理流程。
- 收到符合條件的群組訊息後建立 `pending` 狀態的確認單。
- 使用 LINE push message 對個人發送確認卡片。
- 確認卡片顯示：
  - 股票代號與名稱
  - 買進或賣出
  - 預估價格
  - 預估金額
  - 股數或張數
  - 訊號來源
  - 有效期限
- 按鈕：
  - 確認下單
  - 取消

完成標準：

- 群組訊息可以成功產生一筆 `OrderIntent`。
- 個人 LINE 可以收到確認訊息。
- 按下確認或取消後，資料庫狀態會更新。
- 此階段不會送出任何真實委託。

### Phase 2：Paper Trading

加入模擬下單邏輯，但仍不呼叫 Shioaji。

待辦：

- 新增 `paper_submitted`、`paper_filled`、`paper_cancelled` 等狀態。
- 按下確認後，依照當下價格建立模擬委託。
- 將模擬委託寫入交易紀錄。
- LINE 回覆模擬委託結果。

完成標準：

- 使用者可以完整走完「群組訊號 -> 個人確認 -> 模擬下單 -> 結果通知」。
- 可重複測試不同訊號，不影響真實帳戶。

### Phase 3：Shioaji Simulation

串接 Shioaji 模擬環境。

待辦：

- 建立 Shioaji adapter。
- 建立下單服務，例如 `brokerOrderService.js` 或獨立 Python/HTTP service。
- 支援股票委託欄位：
  - `action`: Buy / Sell
  - `price`
  - `quantity`
  - `price_type`: LMT / MKT
  - `order_type`: ROD / IOC / FOK
  - `order_lot`: Common / Odd / IntradayOdd
  - `order_cond`: Cash
- 回寫 Shioaji 回傳的：
  - order id
  - seqno
  - ordno
  - status
  - status code
  - message
- 加入查詢委託狀態與成交狀態的排程或手動 API。

完成標準：

- 確認後會送出 Shioaji simulation order。
- 委託成功、失敗、部分成交等狀態都能被記錄。
- LINE 會通知委託結果。

### Phase 4：正式下單前風控

正式下單前必須完成風控與權限控管。

必要防呆：

- 只接受指定群組 ID 的訊號。
- 只接受指定老師或指定 sender marker 的訊號。
- 只允許指定個人 userId 確認下單。
- 確認單必須有有效期限，例如 60 秒。
- 過期確認單不可下單。
- 同一則 LINE message 不可重複下單。
- 同一個 `OrderIntent` 只能確認一次。
- 下單前重新查價。
- 若價格偏離訊號價格超過設定門檻，要求重新確認。
- 單筆最大金額限制。
- 每日最大下單金額限制。
- 股票黑名單。
- 非交易時段禁止市價單。
- Shioaji 失敗時不可自動無限重試。

完成標準：

- 任何重送 webhook、重複點擊、過期點擊都不會造成重複下單。
- 風控失敗時，系統會清楚通知原因。

### Phase 5：正式下單

在 paper trading 與 simulation 穩定後，才開啟正式下單。

待辦：

- 使用環境變數控制模式：
  - `ORDER_MODE=disabled`
  - `ORDER_MODE=paper`
  - `ORDER_MODE=simulation`
  - `ORDER_MODE=live`
- 正式模式預設關閉。
- 正式下單前 LINE 確認訊息必須明確標示「正式下單」。
- 建立每日交易摘要。
- 建立錯誤告警。

完成標準：

- live 模式只能透過環境變數開啟。
- 每筆正式委託都有完整 audit trail。
- 可追蹤訊號來源、確認者、確認時間、委託結果與成交結果。

## 建議資料模型

### OrderIntent

```js
{
  source: 'line_group',
  groupId: String,
  roomId: String,
  sourceMessageId: String,
  sourceEventTimestamp: Date,

  approverUserId: String,
  approverDisplayName: String,

  code: String,
  name: String,
  action: 'buy' | 'sell',
  tradeType: 'buy' | 'sell_half' | 'sell_all',

  priceType: 'MKT' | 'LMT',
  signalPrice: Number,
  confirmPrice: Number,
  limitPrice: Number,
  quantity: Number,
  amount: Number,

  status: 'pending' | 'confirmed' | 'cancelled' | 'expired' | 'submitted' | 'failed' | 'filled',
  expiresAt: Date,
  confirmedAt: Date,
  cancelledAt: Date,

  rawText: String,
  riskChecks: Array,

  broker: 'shioaji',
  brokerMode: 'paper' | 'simulation' | 'live',
  brokerOrderId: String,
  brokerSeqNo: String,
  brokerOrdNo: String,
  brokerStatus: String,
  brokerStatusCode: String,
  brokerMessage: String,
  errorMessage: String
}
```

建議 unique index：

```js
{ source: 1, sourceMessageId: 1, code: 1, action: 1, tradeType: 1 }
```

如果一則訊息可能產生多筆同股票同方向委託，需再加 `sequence` 欄位。

## LINE 確認訊息範例

```text
待確認委託

股票：2303 聯電
方向：買進
委託：市價
預估價格：52.3
股數：1 張 / 1000 股
預估金額：52,300
有效期限：60 秒

[確認下單] [取消]
```

按鈕建議使用 postback action：

```text
action=confirm_order&orderIntentId=...
action=cancel_order&orderIntentId=...
```

後端收到 postback 後，必須以資料庫中的 `OrderIntent` 為準，不可只相信 LINE postback 送回的文字內容。

## LINE 訊息額度策略

為了節省 LINE 官方帳號的推播額度，系統預設採用「必要時才 push，其餘盡量 reply」的策略。

### 使用 push message 的情境

- 群組收到交易訊號後，推送個人確認單。
- 委託或成交結果不是由使用者當下操作觸發，且需要即時通知。
- 重大錯誤或風控警示，需要主動通知使用者。

群組訊號要送到個人聊天室時，必須使用 push message，因為 webhook 的 reply token 只能回覆原本觸發事件的聊天室。也就是說，群組訊息的 reply 只會回到群組，不能轉成個人確認訊息。

### 使用 reply message 的情境

- 使用者按下「確認下單」後，回覆「已送出委託」或「風控擋下」。
- 使用者按下「取消」後，回覆「已取消」。
- 使用者主動輸入「待確認」後，回覆目前 pending orders。
- 使用者主動查詢委託狀態或今日摘要。

reply message 不消耗推播額度，因此確認、取消、查詢這類由使用者主動觸發的互動，應優先使用 reply message。

### 省額度模式

如果未來推播額度不足，可以加入省額度模式：

```text
群組收到訊號
  -> 只建立 pending order，不主動推播

使用者在個人聊天室輸入「待確認」
  -> bot 用 reply message 回覆所有待確認單
  -> 使用者按確認
  -> bot 用 reply message 回覆結果
```

此模式幾乎不消耗 push 額度，但缺點是使用者不會即時收到確認通知。正式交易預設仍建議保留「確認單 push」，其他結果盡量用 reply 或彙整通知。
## Shioaji 串接建議

Node.js 後端可以用兩種方式串接：

1. 呼叫 Shioaji HTTP service。
2. 建立獨立 Python broker service，由 Node.js 透過 HTTP 呼叫。

建議優先採用獨立 broker service，原因：

- Shioaji 主要使用情境偏 Python。
- 登入、憑證、委託狀態更新比較適合由長時間運行的服務管理。
- 未來切換 simulation / live 較清楚。
- Node.js LINE webhook 可以保持輕量，只負責接訊息與狀態流轉。

## 狀態機

```text
pending
  -> confirmed
  -> submitted
  -> filled

pending
  -> cancelled

pending
  -> expired

confirmed
  -> failed

submitted
  -> failed
```

重要規則：

- `pending` 才能確認。
- `expired`、`cancelled`、`submitted`、`filled` 不可再次確認。
- 確認下單前必須重新讀取資料庫最新狀態。
- 下單與狀態更新要盡量做到 idempotent。

## 風控設定草案

可放在環境變數或資料庫設定中：

```env
ORDER_MODE=paper
ORDER_APPROVER_USER_ID=Uxxxxxxxx
ORDER_CONFIRM_EXPIRES_SECONDS=60
ORDER_MAX_SINGLE_AMOUNT=100000
ORDER_MAX_DAILY_AMOUNT=300000
ORDER_MAX_PRICE_DEVIATION_PERCENT=2
ORDER_ALLOWED_GROUP_IDS=Cxxxxxxxx
ORDER_BLOCKED_SYMBOLS=00878,0050
```

## 需要注意的技術點

- LINE push message 需要知道個人 userId，且使用者需能接收官方帳號訊息。
- LINE postback 會回到同一個 webhook，因此 `handleEvent` 需要支援 `event.type === 'postback'`。
- 目前 `TradeJournalEntry` 對 `{ platform, messageId }` 做 unique index，若一則 LINE 訊息解析出多筆交易，未來需要調整 idempotency 設計。
- 不建議在 webhook request 期間直接做耗時下單流程；可以先建立狀態，再交給服務處理。
- 所有正式下單相關 secret、憑證、CA 都不可進 Git。
- 正式下單前要先跑 paper trading 與 Shioaji simulation。

## 建議優先順序

1. 新增 `OrderIntent` model。
2. 新增 postback handler。
3. 新增個人確認 Flex Message。
4. 完成 pending / confirmed / cancelled / expired 狀態。
5. 加入 paper trading。
6. 加入風控檢查。
7. 串接 Shioaji simulation。
8. 建立委託狀態查詢。
9. 加入正式下單模式開關。
10. 小金額 live 測試。

## 成功標準

這個功能完成後，系統應該能做到：

- 群組訊號不會直接造成下單。
- 每筆下單都需要個人確認。
- 每筆確認都有期限、權限與風控檢查。
- 每筆委託都有完整紀錄。
- 發生錯誤時不會重複下單。
- 可以從 paper trading 平滑升級到 simulation，再升級到 live。
